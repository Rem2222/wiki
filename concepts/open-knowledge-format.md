# Open Knowledge Format (OKF)

**Версия:** 0.1 (Draft, 2026)
**Автор:** Google Cloud Platform — Knowledge Catalog
**Cтатус:** черновик спецификации

OKF — это открытый, человеко- и агент-дружественный формат для представления **знаний**: метаданных, контекста и курированной информации о данных и системах. Предназначен для авторства людьми, генерации агентами, обмена между организациями и потребления и теми, и другими.

Главный принцип: формат намеренно минимален. Если вы умеете `cat` — вы можете читать OKF. Если умеете `git clone` — можете распространять.

---

## 1. Мотивация

Пространство представления знаний для AI-агентов развивается быстро, и появляется много несовместимых конвенций. OKF исходит из того, что знания лучше всего представлять в общедоступных, устоявшихся форматах:

- **Читаемые** человеком без специальных инструментов
- **Парсимые** агентами без bespoke SDK
- **Diff-уемые** в системе контроля версий
- **Портируемые** между инструментами, организациями и временем

### Цели

1. Универсальный формат, в который **агенты-обогатители** могут писать
2. Инструктировать **агентов-потребителей**, как читать и обходить знания
3. Упростить **обмен** знаниями между системами и организациями
4. Стандартизировать минимальный набор **обязательных полей**

### Не-цели

- Фиксированная таксономия типов концептов
- Предписание инфраструктуры хранения, serving'а или запросов
- Замена предметно-ориентированных схем (Avro, Protobuf, OpenAPI) — OKF на них **ссылается**, но не замещает

## 2. Терминология

- **Knowledge Bundle** (пакет знаний) — самодостаточная иерархическая коллекция документов. Единица распространения.
- **Concept** (концепт) — единица знания внутри пакета. Один markdown-документ.
- **Concept ID** — путь файла концепта внутри пакета без `.md`. Например, `tables/users` для `tables/users.md`.
- **Frontmatter** — YAML-блок метаданных между `---` в начале файла
- **Body** — всё после frontmatter
- **Link** — обычная markdown-ссылка между концептами
- **Citation** — ссылка из концепта на внешний источник

---

## 3. Структура пакета

```
path/to/bundle/
├── index.md                      # Опционально. Каталог для progressive disclosure
├── log.md                        # Опционально. Хронологический журнал изменений
├── <concept>.md                  # Концепт в корне пакета
└── <subdirectory>/
    ├── index.md
    ├── <concept>.md
    └── <subdirectory>/
        └── …
```

Распространяется как: git-репозиторий (рекомендуется), tarball/zip, или поддиректория внутри большого репозитория.

### Зарезервированные имена

| Файл        | Назначение                  |
|---|---|
| `index.md`  | Каталог директории          |
| `log.md`    | История изменений           |

## 4. Структура документа концепта

Каждый концепт — UTF-8 markdown с двумя частями.

### 4.1 Frontmatter

```yaml
---
type: <Type name>                  # REQUIRED
title: <Optional display name>
description: <Optional one-line summary>
resource: <Optional canonical URI>
tags: [<tag>, <tag>, …]
timestamp: <ISO 8601 datetime>
# … любые producer-defined ключи
---
```

**Обязательное:**
- `type` — короткий идентификатор вида концепта (например, `BigQuery Table`, `Playbook`, `Metric`, `Reference`). Типы **не регистрируются централизованно**. Потребитель должен толерантно обрабатывать неизвестные типы.

**Рекомендуемые (в порядке приоритета):**
- `title`, `description`, `resource`, `tags`, `timestamp`

**Расширяемость:** producer может добавлять любые ключи. Consumer должен сохранять неизвестные ключи при round-trip и не отвергать документ.

### 4.2 Body

Стандартный markdown. Рекомендуется структурный подход: заголовки, списки, таблицы, code blocks.

Конвенциональные секции:
- `# Schema` — описание полей/колонок
- `# Examples` — примеры использования
- `# Citations` — внешние источники

### 4.3 Пример: концепт с привязкой к ресурсу

```markdown
---
type: BigQuery Table
title: Customer Orders
description: One row per completed customer order across all channels.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders, revenue]
timestamp: 2026-05-28T14:30:00Z
---
```

### 4.4 Пример: концепт без привязки к ресурсу

```markdown
---
type: Playbook
title: Incident response — data freshness alert
description: Steps to triage a freshness alert on the orders pipeline.
tags: [oncall, incident]
timestamp: 2026-04-12T09:00:00Z
---
```

## 5. Cross-linking

Две формы:
- **Абсолютные (bundle-relative):** `/tables/customers.md` — рекомендуемая форма
- **Относительные:** `./other.md`

Семантика: ссылка от A к B утверждает *отношение*. Конкретный тип отношения передаётся окружающим текстом, а не самой ссылкой.

Потребители **должны толерантно** обрабатывать битые ссылки — цель может просто ещё не написана.

---

## 6. Index Files

`index.md` может быть в любой директории. Без frontmatter. Формат:

```markdown
# Section Name

- [Title](relative-url) - description
```

Позволяет progressive disclosure — агент или человек видит, что доступно, не открывая каждый документ.

---

## 7. Log Files

Опциональный `log.md`. Хронология, newest first:

```markdown
## 2026-05-22
* **Update**: Added new BigQuery table reference
* **Creation**: Established the Dataplex Playbook
```

Даты в `YYYY-MM-DD`.

---

## 8. Citations

Под заголовком `# Citations` внизу документа, нумерованные ссылки на внешние источники.

---

## 9. Конформность (Conformance)

Пакет конформен OKF v0.1 если:
1. Каждый non-reserved `.md` файл содержит парсируемый YAML frontmatter
2. Каждый frontmatter содержит непустое поле `type`
3. Каждый `index.md` и `log.md` следует своей структуре

Потребитель **не должен** отвергать пакет из-за: отсутствия опциональных полей, неизвестных `type`, неизвестных ключей frontmatter, битых cross-links, отсутствующих `index.md`.

---

## 10. Связь с другими форматами

OKF близок к нескольким устоявшимся паттернам:
- **LLM-вики** (markdown + frontmatter как agent-readable knowledge base)
- **Obsidian / Notion** (иерархический markdown с cross-links)
- **Metadata-as-code** (каталоговые метаданные рядом с кодом, а не в отдельном registry)

Отличие OKF — он **специфицирован**: зафиксирован минимальный набор правил для интероперабельности без диктата инструментария.

---

## 11. Версионирование

Формат: `<major>.<minor>`.
- **Minor** — обратно-совместимые дополнения
- **Major** — breaking changes

Пакет может объявить версию через `okf_version: "0.1"` в frontmatter корневого `index.md`.

## Идея на апгрейд wiki

Текущая wiki Романа уже очень близка к OKF:

| OKF | Текущая wiki |
|---|---|
| `type: <Type name>` — REQUIRED | `type` есть в SCHEMA.md, но не на всех страницах и не контролируется |
| `index.md` с перечислением | Есть корневой `index.md`, поддиректории не имеют своих index |
| `log.md` с датами | Есть корневой `log.md`, формат близкий |
| Cross-links `wikilinks` | Используются активно |
| YAML frontmatter | Есть, но без `timestamp` и `resource` на многих страницах |

**Что можно улучшить:**
1. **Добавить `type` на все страницы** — сейчас `type` описан в SCHEMA.md, но не все страницы его содержат. OKF требует `type` обязательно.
2. **Гарантировать `timestamp`** — OKF рекомендует ISO 8601 дату последнего изменения.
3. **index.md в поддиректориях** — для быстрого просмотра агентом без сканирования всей папки.
4. **Даты в log.md** — перевести на ISO 8601 `YYYY-MM-DD` для единообразия с OKF.
5. **Проверка на конформность** — скрипт, который проверяет все страницы на наличие обязательных полей (аналог OKF conformance check).

OKF мог бы стать формальным стандартом, под который подогнать wiki — это упростило бы обмен знаниями с другими агентами и системами в будущем.
