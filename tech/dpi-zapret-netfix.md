# DPI, Zapret, TgWsProxy и NetFix

---

## Что такое DPI (Deep Packet Inspection)

DPI — технология «глубокого анализа пакетов», которую интернет-провайдеры (и вышестоящие ТСПУ) используют для:

- **Блокировки** сайтов по URL, IP, SNI, протоколу
- **Замедления** (throttling) — например, Telegram или YouTube
- **Обнаружения** VPN/прокси по сигнатурам протоколов (WireGuard, OpenVPN и т.д.)

DPI работает **не как брандмауэр** (который просто смотрит IP/порт), а анализирует содержимое пакетов. Он может:

- Читать SNI из TLS-рукопожатия (какой сайт открываешь по HTTPS)
- Распознавать HTTP-запросы и заголовки (Host, User-Agent)
- Идентифицировать протоколы по сигнатурам (MTProto, WireGuard, QUIC)
- Инжектить RST-пакеты, разрывающие соединение
- Модифицировать трафик (редирект на заглушку)

**Два типа DPI:**

| Тип | Описание |
|-----|----------|
| **Пассивный** | Читает трафик, может инжектить свои пакеты (RST), но не блокирует проходящие пакеты |
| **Активный** | Стоит в разрыве провода. Может дропать любые пакеты, распознавать TCP-потоки и блокировать их |

В России сейчас везде **активный DPI ТСПУ** (Технические Средства Противодействия Угрозам). Простые iptables-правила уже не работают.

## Как работает обход DPI

Идея: **послать то, на что DPI не рассчитывает, и что ломает ему алгоритм распознавания**. DPI вынужден обрабатывать огромный трафик на ограниченных ресурсах — он интерпретирует пакеты лишь ограниченно, и этим можно пользоваться.

### Основные методы обхода

**1. TCP-сегментация (splitting)**
HTTP-запрос `GET / HTTP/1.1\r\nHost: example.com...` отправляется не одним куском, а двумя: сначала `GET`, потом `/ HTTP/1.1\r\nHost: example.com...`. DPI не может распознать начало запроса и пропускает.

**2. Фейки (fake packets)**
Перед реальным запросом отправляется «мусорный» пакет с поддельным HTTP-запросом или TLS ClientHello. DPI видит его, принимает за легитимный, а реальный трафик проходит незамеченным.

**3. Десинхронизация sequence numbers**
Изменение TCP sequence numbers в пакетах так, что DPI теряет синхронизацию с потоком и не может корректно анализировать трафик.

**4. Манипуляция TTL**
Изменение TTL в поддельных пакетах так, чтобы они доходили до DPI (который стоит в разрыве), но НЕ доходили до сервера. Сервер видит только чистый трафик.

**5. Изменение регистра Host:**
`Host: example.com` → `host: example.com` или `HoSt: example.com`. Некоторые DPI ищут заголовок строго в нижнем регистре.

**6. Добавление точки в конце имени хоста**
`Host: example.com.` — корректный DNS-синтаксис, но многие DPI на это не рассчитывают.

**7. IP-фрагментация**
Разбивка IP-пакета на фрагменты, которые DPI не может собрать обратно для анализа.

**8. UDP-удлинение (udplen)**
Добавление мусора в UDP-пакеты (например, для Discord голосовых чатов).

## Zapret

**Репозиторий:** [bol-van/zapret](https://github.com/bol-van/zapret) / [zapret2](https://github.com/bol-van/zapret2)  
Автор: bol-van | **Цель:** автономное средство противодействия DPI без сторонних серверов

Zapret — это набор утилит, которые модифицируют трафик на уровне пакетов, не давая DPI распознать «запрещённые» запросы.

### Компоненты

**nfqws** — модификатор пакетов для Linux (через NFQUEUE):
- Перехватывает пакеты в очереди netfilter
- Применяет к ним стратегии десинхронизации (сегментация, фейки, манипуляция TTL)
- Отправляет модифицированные пакеты дальше
- Работает на уровне ядра — минимальная задержка

**tpws** — transparent proxy для TCP:
- Проксирует TCP-соединения
- Может модифицировать HTTP-запросы на лету (регистр Host, сегментация, добавление пробелов)
- Работает на уровне пользовательского пространства (медленнее nfqws, но проще)

**winws** — версия nfqws для Windows:
- Использует **WinDivert** (аналог NFQUEUE для Windows) — драйвер для перехвата пакетов
- Именно этот компонент запускает NetFix под капотом

### Стратегии

Zapret использует разные «стратегии» (наборы параметров), потому что DPI у разных провайдеров отличаются:

- `general` — базовая TCP-сегментация
- `general_ALT` — альтернативная сегментация
- `general_FAKE` — с фейковыми пакетами
- И другие (подбираются экспериментально)

### Ограничения

- Не работает, если блокировка по IP (не по домену/протоколу)
- Не работает, если соединение проходит через полноценный прокси/сокс (squid)
- Требует права администратора (драйвер WinDivert)
- Стратегии могут переставать работать — DPI адаптируется

## TgWsProxy

**Репозиторий:** [Flowseal/tg-ws-proxy](https://github.com/Flowseal/tg-ws-proxy)  
**★:** 7181 | **Цель:** ускорение Telegram Desktop через локальный MTProto прокси

TgWsProxy — локальный MTProto прокси-сервер, который решает **конкретную проблему замедления Telegram**. В отличие от Zapret (который работает на уровне пакетов), TgWsProxy:

1. Поднимает локальный MTProto-прокси на 127.0.0.1:...
2. Telegram Desktop подключается к нему как к обычному MTProto прокси
3. Прокси пересылает трафик через WebSocket (или другой транспорт)
4. DPI не может заблокировать MTProto по сигнатуре, так как трафик идёт внутри WebSocket

**Разница с Zapret:**
- Zapret работает со всем трафиком системы (сетевая фильтрация пакетов)
- TgWsProxy работает только с Telegram (прикладной прокси для одного приложения)
- Они не конфликтуют — могут использоваться одновременно

## NetFix — GUI-обёртка

**Репозиторий:** [rupleide/NetFix](https://github.com/rupleide/NetFix)  
**Автор:** rupleide | **Статья на Хабре:** [habr.com/ru/articles/1044292](https://habr.com/ru/articles/1044292/)  
**Стек:** C# / .NET 8.0 / WPF | **Дата:** 6 июня 2026

NetFix — не инструмент обхода DPI, а **GUI-менеджер**, который автоматизирует запуск и мониторинг Zapret и TgWsProxy.

### Что делает NetFix под капотом

1. **One-click запуск:**
   - Проверяет, есть ли бинарники Zapret (`winws.exe`) и TgWsProxy
   - Если нет — скачивает с GitHub Releases (AutoDownloadService)
   - Запускает процессы с нужными параметрами
   - Проверяет, что они реально работают (сетевая диагностика)

2. **Диагностика (DiagnosticsEngine):**
   - Ping до целевых серверов
   - DNS-резолв (обход подмены DNS)
   - HTTP/HTTPS доступность
   - WebSocket-проверка
   - Speedtest (замер скорости)
   
3. **Управление конфигами Zapret:**
   - Визуальный редактор стратегий
   - Тестирование разных стратегий без ручного редактирования батников
   - Сохранение работающей конфигурации

4. **Процесс-менеджмент:**
   - Мониторинг состояния Zapret и TgWsProxy (зелёный/жёлтый/красный индикатор)
   - Auto-restart при падении
   - Graceful shutdown при закрытии

5. **На старте:**
   - Проверка бинарей → скачивание → запуск → проверка соединения → отображение статуса

### Технические детали реализации

- Единственная NuGet-зависимость: `DiscordRichPresence` (показывает статус в Discord)
- Встроенные средства: `HttpClient` (вместо RestSharp), `System.Text.Json` (вместо Newtonsoft), `MediaPlayer` (вместо NAudio)
- Ритм-игра как пасхалка (NoteMap.cs)
- Автообновления через UpdateService
- Трей-иконка через WinForms NotifyIcon

## Сравнение методов обхода

| Инструмент | Уровень | Что обходит | Сложность | Нужен VPS/сервер |
|-----------|---------|-------------|-----------|------------------|
| **Zapret** | Пакетный (ядро) | DPI провайдера (весь трафик) | Средняя (батники, конфиги) | Нет |
| **TgWsProxy** | Прикладной (MTProto) | Замедление Telegram | Низкая | Нет |
| **VPN (WireGuard/OpenVPN)** | Транспортный | Всё (если IP не заблокирован) | Низкая | Да |
| **NaiveProxy** | Прикладной (HTTP/2) | DPI + блокировка по IP | Средняя | Да |
| **Hysteria** | Транспортный (QUIC) | DPI + потеря пакетов | Средняя | Да |
| **Tor** | Цепочка прокси | Анонимность + цензура | Низкая | Нет |
